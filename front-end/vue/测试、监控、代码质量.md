# 工程质量与研发流程基础整理

## 1. 单元测试基础

单元测试的目标是：**验证最小功能单元是否正确**。  
这里的“最小单元”通常是一个函数、一个工具方法、一个组件中的一段逻辑。

### 为什么要写单元测试
- 提高代码可靠性
- 降低重构风险
- 提前发现边界问题
- 让核心逻辑更稳定

### 常见测试对象
- 工具函数
- 数据处理函数
- 表单校验逻辑
- 权限判断逻辑
- 状态管理中的纯函数
- 后端 service 层方法

### 不适合优先写单元测试的地方
- 纯静态页面
- 经常变动的 UI 细节
- 非核心展示代码

### 一个简单例子
```js
function sum(a, b) {
  return a + b
}
```

测试代码：

```js
import { describe, it, expect } from 'vitest'
import { sum } from './sum'

describe('sum', () => {
  it('1 + 2 应该等于 3', () => {
    expect(sum(1, 2)).toBe(3)
  })
})
```

### 单元测试的核心原则

#### 1）只测一个点

一个测试用例最好只验证一个行为。

#### 2）要覆盖正常情况和异常情况

比如：

* 正常输入
* 空值
* 非法值
* 边界值

#### 3）测试要稳定

不能今天过、明天挂。
如果依赖时间、网络、数据库，就要 mock。

### 前端常见测试内容

* 工具函数
* hooks/composables
* 权限函数
* 格式化函数
* API 参数转换逻辑

### 后端常见测试内容

* service 层
* 参数校验逻辑
* 权限校验逻辑
* 业务计算逻辑

## 2. 接口测试基础

接口测试的目标是：**验证 API 是否符合预期**。

它关注的是：

* 路由是否通
* 参数是否合法
* 返回值是否正确
* 状态码是否符合预期
* 鉴权是否生效

### 接口测试和单元测试的区别

#### 单元测试

测一个函数或一个最小逻辑单元。

#### 接口测试

测一个完整 API 请求流程。

### 常见接口测试内容

* 登录接口
* 注册接口
* 用户信息查询接口
* 新增/编辑/删除接口
* 文件上传接口
* 权限接口

### 要测试哪些维度

#### 1）正常流程

例如：

* 用户名密码正确时能登录成功

#### 2）异常流程

例如：

* 密码错误
* 参数缺失
* token 无效
* 权限不足

#### 3）边界情况

例如：

* 分页页码为 0
* 上传文件为空
* 文本长度超出限制

### 常见工具

* Postman
* Apifox
* Swagger
* Supertest（Node 项目里常见）

### Node 接口测试示例

```js
import request from 'supertest'
import app from '../app'

describe('登录接口测试', () => {
  it('用户名密码正确时应返回 200', async () => {
    const res = await request(app)
      .post('/api/login')
      .send({
        username: 'admin',
        password: '123456'
      })

    expect(res.status).toBe(200)
    expect(res.body.code).toBe(0)
  })
})
```

### 接口测试的价值

* 保证核心接口可用
* 降低联调风险
* 让重构更安心
* 方便回归测试

## 3. 错误监控

错误监控的目标是：**系统出错时，能被及时发现、定位、追踪**。

### 为什么要做错误监控

很多问题在本地测不出来，只会在真实用户环境出现，比如：

* 某个浏览器兼容问题
* 某个页面白屏
* 某个接口线上超时
* 某个用户特定操作导致报错

### 前端常见错误类型

* JS 运行时报错
* Promise 未捕获异常
* 资源加载失败
* 页面白屏
* 接口请求失败
* 路由跳转错误

### 后端常见错误类型

* 接口 500
* 数据库连接失败
* Redis 连接失败
* 参数错误
* 第三方服务异常
* 任务执行异常

### 一个基础思路

发生错误时，记录这些信息：

* 错误时间
* 页面路径
* 用户 ID
* 浏览器信息
* 错误堆栈
* 请求参数
* 响应结果
* 环境信息

### 前端常见做法

* `window.onerror`
* `unhandledrejection`
* 请求拦截器中统一记录异常
* 接入监控平台

### 后端常见做法

* 全局异常处理中间件
* 统一日志输出
* 接口错误统计
* 告警通知

### 错误监控的核心价值

不是“记录报错”这么简单，而是：

* 能不能快速知道错了
* 能不能快速找到原因
* 能不能评估影响范围

## 4. 日志思维

日志不是“随便打印点东西”，而是帮助你定位问题、追踪流程、分析行为的关键手段。

### 为什么日志很重要

当线上出问题时，你看不到用户操作过程，
日志就是你还原现场的重要依据。

### 常见日志分类

#### 1）访问日志

记录谁访问了什么接口、什么页面。

#### 2）业务日志

记录关键业务动作，例如：

* 用户登录
* 创建订单
* 修改权限
* 删除数据

#### 3）错误日志

记录异常信息、堆栈、上下文。

#### 4）安全日志

记录敏感行为，例如：

* 登录失败
* 修改密码
* 权限变更
* 异常访问

### 好日志应该具备什么

* 关键信息完整
* 结构清晰
* 能搜索
* 能定位问题
* 不泄露敏感数据

### 不好的日志

```js
console.log('出错了')
```

### 好一点的日志

```js
console.error({
  msg: '用户登录失败',
  userId: 1001,
  time: '2026-04-21 10:00:00',
  reason: '密码错误',
  ip: '127.0.0.1'
})
```

### 日志思维的关键

不是打印越多越好，而是打印得有价值。

你至少要知道：

* 哪个模块出错了
* 谁操作的
* 什么时候出的错
* 输入了什么
* 返回了什么
* 错误影响到哪一步

## 5. 性能指标

性能指标的目标是：**衡量系统快不快、稳不稳、用户体验好不好**。

### 前端常见性能指标

#### 1）首屏加载时间

页面从开始加载到首屏内容能看到的时间。

#### 2）接口响应时间

请求发出去到结果返回的耗时。

#### 3）资源加载时间

JS、CSS、图片等资源加载速度。

#### 4）页面交互流畅度

用户点击、输入、滚动时是否卡顿。

### 前端关注点

* 首屏是否过慢
* 打包体积是否过大
* 图片是否过大
* 是否有阻塞渲染
* 是否有重复请求
* 是否有内存泄漏

### 后端常见性能指标

#### 1）接口耗时

一个请求平均多久返回。

#### 2）吞吐量

单位时间内能处理多少请求。

#### 3）错误率

失败请求比例。

#### 4）CPU / 内存使用率

服务负载是否异常。

#### 5）数据库响应时间

SQL 是否慢。

### 你要形成的基本性能意识

* 慢，不一定是前端问题
* 卡，也不一定是后端问题
* 性能排查要分层看：

  * 页面渲染
  * 网络请求
  * 服务端处理
  * 数据库查询

### 常见优化方向

#### 前端

* 懒加载
* 代码分包
* 图片优化
* 减少重复渲染
* 减少大体积依赖
* 缓存静态资源

#### 后端

* SQL 优化
* Redis 缓存
* 连接池
* 异步任务
* 限流
* 分页查询

## 6. 项目文档

项目文档的目标是：**让别人能看懂、能接手、能运行、能维护**。

### 为什么项目文档重要

很多项目做完后没人看得懂：

* 环境怎么配不知道
* 启动命令不知道
* 接口规范不知道
* 项目结构看不懂
* 部署流程没人会

这种项目很难协作，也很难维护。

### 项目文档至少应该包含什么

#### 1）项目介绍

这个项目是做什么的。

#### 2）技术栈

用了哪些核心技术。

#### 3）目录结构

每个目录大概干什么。

#### 4）安装与启动

如何安装依赖、配置环境、运行项目。

#### 5）环境变量说明

每个环境变量有什么作用。

#### 6）接口说明

接口地址、参数、返回值、鉴权方式。

#### 7）部署说明

如何打包、如何上线。

#### 8）常见问题

启动失败怎么办，数据库怎么初始化。

### README 示例结构

```md
# 项目名称

## 项目介绍

## 技术栈

## 安装依赖
npm install

## 启动项目
npm run dev

## 打包
npm run build

## 环境变量说明

## 项目目录结构

## 接口文档地址

## 部署说明

## 常见问题
```

### 好文档的特点

* 新人能快速上手
* 运维能看懂部署
* 面试官能看懂你的项目成熟度
* 未来的你自己也能看懂

## 7. Git 工作流

Git 工作流的目标是：**让团队多人协作开发更有秩序**。

### 为什么不能乱提交

如果大家都直接往主分支推代码，容易出现：

* 代码互相覆盖
* 功能混在一起
* 回滚困难
* 上线风险大

### 常见分支

#### main / master

主分支，通常用于正式环境。

#### develop

开发分支，日常开发合并到这里。

#### feature/*

功能分支，例如：

* feature/login
* feature/user-management

#### hotfix/*

线上紧急修复分支。

### 一个常见开发流程

1. 从 develop 拉一个功能分支
2. 在功能分支开发
3. 提交代码
4. 推到远程
5. 发起合并请求
6. 代码 review
7. 合并到 develop
8. 测试通过后再合并到 main

### 常用命令

```bash
# 拉代码
git pull

# 新建并切换分支
git checkout -b feature/login

# 查看状态
git status

# 添加文件
git add .

# 提交
git commit -m "feat: 完成登录功能"

# 推送分支
git push origin feature/login

# 切换分支
git checkout develop

# 合并
git merge feature/login
```

### 提交信息建议

尽量规范一些，例如：

* `feat: 新增登录功能`
* `fix: 修复用户列表分页问题`
* `refactor: 重构权限校验逻辑`
* `docs: 更新部署文档`
* `test: 补充登录接口测试`

### Git 工作流的核心价值

* 降低协作混乱
* 提高代码可追踪性
* 方便 review
* 方便回滚

## 8. CI/CD 基础概念

CI/CD 是现代开发流程里很重要的一部分。

### 8.1 CI 是什么

CI = Continuous Integration，持续集成。

意思是：

* 开发者提交代码后
* 系统自动进行构建、检查、测试
* 尽早发现问题

### 8.2 CD 是什么

CD 一般有两种理解：

#### Continuous Delivery，持续交付

代码通过验证后，随时可以发布。

#### Continuous Deployment，持续部署

代码通过验证后，自动部署到环境。

### CI 常见做的事情

* 安装依赖
* 代码检查
* 单元测试
* 接口测试
* 构建项目
* 打包产物

### CD 常见做的事情

* 发布测试环境
* 发布预发环境
* 发布正式环境
* 重启服务
* 回滚版本

### 一个简单理解

```text
开发提交代码
   ↓
触发流水线
   ↓
安装依赖
   ↓
代码检查
   ↓
执行测试
   ↓
构建项目
   ↓
部署到服务器
```

### CI/CD 的价值

* 减少手工操作
* 减少发布出错
* 提高交付效率
* 提高研发规范性

### 常见平台

* GitHub Actions
* GitLab CI
* Jenkins
* 云厂商流水线

### 一个前端项目的简单 CI/CD 思路

#### CI 阶段

* 安装依赖
* 执行 lint
* 执行测试
* 构建项目

#### CD 阶段

* 打包 Docker 镜像
* 推送镜像仓库
* 服务器拉取新镜像
* 重启容器
